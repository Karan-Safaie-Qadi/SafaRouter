Migration Guide

Upgrade paths between major versions of SafaRouter.

v1.5.x → v2.0.0

What Changed

  • Optional features are now tree-shakeable — unused features don't add to bundle
  • SWR caching for loaders is enabled by default (loaderCache: true)
  • New safa-router/vite plugin for automatic feature detection
  • New safa-router/core for minimal router without optional features
  • New safa-router/typed for TypeScript-safe route definitions
  • New slots for parallel content and intercept for overlay navigation

Migration Steps

No breaking changes — v2.0.0 is fully backward-compatible with v1.5.x.

  1. If you use Vite, add the plugin to your vite.config.js: 
    import { safaRouter } from 'safa-router/vite'
export default { plugins: [safaRouter()] }
  2. For TypeScript projects, optionally migrate to typed routes with safa-router/typed
  3. For minimal bundles, consider switching to safa-router/core if you don't need optional features

Before (v1.5.x)

import { SafaRouter } from 'safa-router'

After (v2.0.0) — same API, no changes needed

import { SafaRouter } from 'safa-router'

// Or for minimal bundle:
import { SafaRouter } from 'safa-router/core'

v1.4.x → v1.5.0

What Changed

  • Per-route meta.hideComponents — hide smart components per route
  • Access allowlist mode — access.mode: 'allowlist' blocks everything except explicitly allowed paths
  • RealtimeManager — SSE/polling/WebSocket hot reload
  • SafaDevServer — standalone dev server class with SPA fallback

Migration Steps

No breaking changes — v1.5.0 is fully backward-compatible with v1.4.x.

  • To use hideComponents, add meta.hideComponents to your route configs
  • To use allowlist mode, set access.mode: 'allowlist' and define access.allowed
  • To enable realtime, add the realtime config to your router
  • For dev server, replace node dev-server.js with your own script using SafaDevServer

v1.3.x → v1.4.0

What Changed

  • Smart Components system (components) — define reusable header/footer components
  • Template resolution — {{ params.x }}, {{ query.x }}, {{ data.x }} in HTML files
  • Auto-resolved page — routes without page resolve from pageDir automatically
  • HTML-first routing is now the primary pattern

Migration Steps

  1. Replace JS page exports with HTML files in your pageDir. Each route expects an .html file matching its path.
  2. Migrate layouts from JS to HTML using the {children} placeholder syntax.
  3. If you used custom rendered layouts, move that logic into the components config as smart components.
  4. No other breaking API changes — all core router APIs remain the same.

v1.2.x → v1.3.0

What Changed

  • New ErrorManager — handle all HTTP status codes with custom error pages
  • New AccessController — block/ignore route patterns with wildcards
  • New Maintenance Mode — toggleable 503 with allowed path bypasses
  • New Route Data Loaders — async data fetching before page render
  • New Route Guards — per-route access control with redirect support
  • New Per-route Transitions — custom CSS transitions per route

Migration Steps

No breaking changes — all existing APIs remain unchanged. New features are opt-in.

  • All new features are optional — your existing config continues to work
  • errorLogging, access, maintenanceMode are new config sections
  • loader, guard, and per-route meta.transition are new route-level options
